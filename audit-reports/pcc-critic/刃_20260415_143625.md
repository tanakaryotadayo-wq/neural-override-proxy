2. 現状理解 (Current Understanding)
ユーザー要求の6項目に関する事実ベースの監査結果：

- **テストカバレッジの実態**
  テストファイル数 / 総ファイル数: 0 / 2 (0%)
  `ms_server.py` (L462-L474): 内部に簡易的な非同期テスト関数 `run_tests()` が存在するのみで、CI等で自動実行できる単体テストファイル（`test_*.py`）は存在しない。

- **importの整合性**
  壊れたimportはないが、依存管理に脆弱性がある。
  `ms_server.py` (L44-L56): `mcp`, `aiohttp` にはトップレベルでの存在チェックがある。
  `ms_server.py` (L257, L281, L316, L333, L363, L453等): `requests` が各ツール関数内で遅延importされているが、トップレベルでのモジュール存在チェックがないため、未インストールの場合は実行時に突然 `ModuleNotFoundError` でクラッシュする。

- **型安全性**
  両ファイルともにPythonのType Hintsは記述されているが、ジェネリクスが不足している。
  `ms_server.py` (L93, L112等): `Dict` や `List` が使用されているが、`Dict[str, Any]` などの内部型の明記がなく、TypeScriptにおける `any` を多用しているのと同等の緩い型定義となっている。
  `pcc_critic.py` (L97, L126等): `-> dict` など組み込みの型ヒントが使用されている。

- **エラーハンドリング**
  `ms_server.py` (L151, L275, L301, L352, L372, L409等): 広範な `except Exception as e:` による全キャッチが多用されている。API通信エラーや構文エラーなどのロジックバグを隠蔽するリスクが高い。
  `pcc_critic.py` (L106, L146, L174): `subprocess.TimeoutExpired` や `json.JSONDecodeError` など、例外クラスが適切にスコープされている。

- **セキュリティ問題**
  ハードコードされた秘密鍵やAPIキーの露出はなし。
  `ms_server.py` (L95-L107): URLやSlack Webhookなどは環境変数から安全に取得している。
  `ms_server.py` (L132): `subprocess.run(safe_cmd)` 実行前に `shutil.which` を用いてバイナリをチェックしており、基本的なコマンドインジェクション対策はされている（`shell=True` も不使用）。

3. 代替案 A / B / C (Alternatives with tradeoffs)
- **案A: テスト・型定義の全面リファクタリング (根本的改善)**
  - 変更内容: `pytest` 導入。全ての `Dict` を `TypedDict` や `Dict[str, Any]` に厳密化。遅延importをトップレベルに移動。
  - トレードオフ: 変更範囲が広大。安全だが実装コストが高く、起動時間が微増するリスク。
- **案B: 例外キャッチの具体化・依存関係チェックの追加 (堅牢性向上)**
  - 変更内容: `Exception` キャッチを `requests.RequestException` などに絞る。トップレベルに `requests` の依存チェックを追加。
  - トレードオフ: 未知の例外でサーバーがクラッシュするリスクが表面化する（フェイルセーフとしては正しい挙動）。
- **案C: 既存コード維持＋E2Eテスト追加 (漸進的改善)**
  - 変更内容: 現状のコードには手を入れず、外部から叩くテストスクリプトだけを追加して現状の振る舞いを固定化。
  - トレードオフ: コストは最小だが、潜在的な型やエラーハンドリングのバグはそのまま残存する。

4. 推奨案 (Recommendation + reasoning)
**案B（例外キャッチの具体化・依存チェック）** を最優先で実施し、その後単体テストを追加するアプローチを推奨する。
理由: MCPサーバー/APIサーバーという性質上、広範な `Exception` キャッチによって外部システム（MLX等）との通信エラーやタイムアウトが握り潰され「何が原因で失敗したか分からない」状態になるのが最大の運用リスクであるため。

5. 実装手順 (Implementation steps)
1. `ms_server.py` のトップレベルに `try: import requests` を追加し、欠如時は起動をブロックする仕様に変更。
2. 各ツールの `except Exception as e:` を、発生しうる具体的な例外（`requests.RequestException`, `asyncio.TimeoutError` 等）に置換。
3. `Dict` を `Dict[str, Any]` 等に修正し、`mypy` での静的型検査をCIに組み込む。
4. `pytest` を導入し、`tests/test_ms_server.py` と `tests/test_pcc_critic.py` を新規作成。モックを活用して各種ツールの動作検証を実装する。

6. テスト観点 (Test perspectives)
- 正常系: MLXへのモックリクエストが200 OKを返した場合に、正しくパースして応答するか。
- 異常系: MLXサーバーがダウンしている、またはタイムアウト（`TimeoutExpired`）した場合に、サーバー全体がクラッシュせず適切なエラーJSONを返すか。
- セキュリティ系: `run_subprocess` に不正なコマンド文字列を渡した場合に、確実に弾かれるか。

7. 残る不確実性 (Remaining uncertainties)
- RyotaOS Runtime (MLX) が予期せぬフォーマットのJSONやHTMLエラーを返した場合、`resp.json()` が `JSONDecodeError` を起こすリスクがある（Assumption: MLXは常に正しいJSONを返すと仮定されている）。
- `mcp`, `aiohttp`, `requests` 等のバージョン指定がないため、将来的なライブラリの破壊的変更で突然動作しなくなるリスク。依存関係のロック（`requirements.txt` のピン留め等）の有無がコード上からは確認できない。
